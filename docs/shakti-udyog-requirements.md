## 15. .NET Core API and Swagger

### API Design Rules

- Build versioned REST endpoints, starting with `/api/v1/`.
- Keep controllers thin; place rules in application services and validate requests before processing.
- Apply authorization policies on every non-public endpoint—never rely only on hidden frontend buttons.
- Return consistent error responses with a trace ID, without exposing internal exceptions.
- Paginate catalogue, enquiry, order, user, and audit-log endpoints.
- Filter all customer queries by the authenticated user’s authorized company. Customers must never be able to guess another customer’s record ID and retrieve data.

## 16. Security, Approval, and Audit Rules

```yaml
security_requirements:
  access_control:
    - Enforce role and permission checks in the .NET API for every protected action.
    - Use least-privilege defaults; a new user has no customer-company access until approved.
    - Require administrator confirmation for role changes, account activation, and sensitive document visibility.
  authentication:
    - Hash passwords using the framework's current secure password hasher.
    - Use short-lived access tokens, secure refresh-token rotation, and server-side revocation.
    - Require multi-factor authentication for administrators; offer it to all staff and customers.
    - Rate-limit login, password reset, RFQ, and upload endpoints.
  files:
    - Check file extension, MIME type, actual file signature, size, and malware status.
    - Store files outside the public web root with encrypted private storage.
    - Authorize every download against the user's company, role, and document visibility setting.
    - Use expiring signed links only when needed; never use permanent public drawing URLs.
  audit:
    - Record who changed what, when, and from which IP/device context for sensitive actions.
    - Preserve audit logs separately from editable operational records.
  data_protection:
    - Use HTTPS everywhere and encrypt sensitive data at rest where appropriate.
    - Back up the database and files on a tested schedule.
    - Provide privacy notice, consent record, data-retention rules, and account-deletion handling.
```

## 17. Workflow Definitions

```text
Customer submits RFQ
  → Data Updater validates details
  → Admin / authorized sales approver reviews
  → Quotation issued to customer
  → Customer accepts or declines
  → Admin confirms order creation
  → Data Updater records production milestones
  → Customer receives status and approved documents
  → Invoice issued
  → Payment status recorded or online payment completed
  → Order dispatched, delivered, and closed
```

### Content Publishing Workflow

```text
Data Updater creates or edits a draft
  → Admin reviews the change
  → Admin approves or rejects with a comment
  → Approved content is published
  → System records an audit entry
```

## 21. Jira Board and Project Management Integration

### Why Jira Helps

Jira gives Shakti Udyog a single internal view of who owns each RFQ, quotation, order task, technical review, quality check, and content update. It is particularly useful when several people are involved and management needs to identify waiting items or delivery risks early.

Jira should be an **internal work-management system**, not the source of truth for customer account data, quotations, orders, or controlled documents. SQL Server and the customer portal remain the official business system; Jira receives only the work references and information required to complete assigned tasks.
